Aegis Help & Documentation

Aegis is a flexible, trigger-based static analysis engine designed for Universal Code Compliance. Unlike traditional linters restricted to a single language, Aegis uses a powerful conditional engine to validate any text-based project‚Äîincluding Java, Python, TypeScript, Docker, Terraform, TIBCO, and MuleSoft.

It allows you to define custom architecture rules, security policies, and best practices using a simple YAML configuration, adapting dynamically to the specific technologies detected in your repository.

üìã What Aegis Validates

Aegis provides deep analysis across a wide spectrum of technologies:

  • Universal: Generic file existence, naming conventions, and regex text patterns.
  • Java/Maven: pom.xml dependencies, folder structures, and checkstyle integration.
  • Python: requirements.txt analysis and script content validation.
  • Cloud/DevOps: Dockerfile best practices, Kubernetes manifests, and Terraform compliance.
  • Integration (MuleSoft/TIBCO): Deep XML inspection, DataWeave scripts, and flow logic validation.
  • Security: Secret scanning, token detection, and sensitive data exposure checks.

‚öôÔ∏è Flexible Project Structure

How Aegis Works

  • Marker Detection: Aegis scans your files to determine project types based on key markers (e.g., `pom.xml`, `requirements.txt`, `package.json`).
  • Configurable Patterns: Configuration folders (like `_config`) are just a default pattern. You can customize the `namePattern` in `rules.yaml` to match any naming convention your organization prefers (e.g., `config`, `settings`, or even root-level files).
  • Rule Scoping: Validation rules are applied intelligently based on these configured markers, ensuring that specific rules only run on relevant project types.

üí° Pro Tip: You define the project markers in rules.yaml using the `markerFiles` configuration. This gives you complete control over how Aegis identifies and validates your unique repository structures. See the Project Identification section in the Rule Guide.

üìö Supported Validation Checks

Aegis supports a wide range of validation types to ensure the integrity and quality of your applications. Possible checks include:

  • Structure & configuration: Validate XML flows, configuration files, global configurations, and file structures.
  • Dependencies: Verify Maven pom.xml, Node.js package.json, and Python requirements.txt.
  • Code Content: Search for specific tokens, patterns, or forbidden content within Source Source Source source files.
  • XML Validation: Perform deep inspection using XPath, check for specific attributes, and validate element content.
  • Property Management & Resolution:
    • Validation: Check for mandatory properties and secure handling.
    • Resolution: üöÄ **Flagship Feature**: Aegis can resolve ${property} placeholders in XML and JSON files using values from your .properties or .yaml config files before validating them.
  • Custom Logic: Define custom checks for JSON validation, generic token searches (Universal Text Scanning), and more to cover any specific organizational requirement.

‚ö†Ô∏è Configuration Tip: Always verify rule-specific parameters in the Rule Guide. For example, GENERIC_TOKEN_SEARCH uses isRegex for pattern matching, whereas matchMode is a global parameter used ONLY for file-level quantifiers (e.g., ALL_FILES vs ANY_FILE).

üîó Git Integration Guide

Aegis supports direct integration with Git providers for automated scanning:

  • Discovery Mode: Browse and select repositories from your GitLab Group or GitHub Organization/User.
  • Manual Mode: Paste any direct Git clone URL (HTTPS).
  • Public Access: Search public groups/users without needing a token (ideal for open-source audits).

Configuration (Settings Page)

  • Tokens: Add your Personal Access Tokens (PAT) in Settings.
    Tip: After clicking "Validate", your settings are auto-saved!
  • Default Group/Org: Set a default Group (GitLab) or Organization (GitHub) to auto-fill the search field on startup.
  • Base URLs: Default to public clouds (gitlab.com, api.github.com) but can be configured for Enterprise instances.

üöÄ How to Use

Command Line

Standard CLI Mode:

java -jar aegis.jar -p /path/to/projects [-c rules.yaml] [-o reports]

GUI Server Mode:

java -jar aegis.jar --gui [--port 8080]

Parameters:
-p, --path: (Required for CLI) Absolute path to the root folder containing projects to scan.
-c, --config: (Optional) Path to a custom rules.yaml file. If omitted, built-in rules are used.
-o, --output: (Optional) Name of the report directory (Default: Aegis-reports).
-g, --gui: Start the Aegis GUI server.
--port: (Optional) Set a custom port for the GUI server (Default: 8080).
-h, --help: Show the help message and exit.
-V, --version: Print version information and exit.

Note: Standalone CLI mode currently supports local folder analysis only. For ZIP/JAR uploads, please use the GUI.

GUI Mode (All Input Modes)

Run start-aegis-gui.bat (Windows) or start-aegis-gui.sh (Mac/Linux)

Opens browser at http://localhost:8080

Supports: Local folder, ZIP upload, JAR upload
